## Why

Brandon-Westman Christmas Cheer (est. 1955) currently runs its donor and donation tracking on an aging Microsoft Access database. The single-user, file-based Access workflow blocks remote access, makes reports cumbersome (the workaround for "one-time vs repeat donor" analysis is to export to Excel), and leaves the charity exposed to the bus-factor of one local copy. We need a foundation we can grow into a multi-user web app — but before any domain modeling or feature work begins, we need a working Rails application skeleton everyone agrees on.

This change does only that: stand up the empty Rails app. No donor data, no UI, no migrations from Access. Future changes layer on top.

## What Changes

- Generate a new Rails 8 application at the repo root using `rails new` with PostgreSQL.
- Configure the Rails 8 built-in authentication generator (`bin/rails generate authentication`) so subsequent changes can gate controllers behind login.
- Pin Ruby and Rails versions via `.ruby-version` and `Gemfile`.
- Add baseline development tooling: RuboCop (rails-omakase), Brakeman, ERB Lint, and the default `bin/dev` Foreman setup that ships with Rails 8.
- Configure the test stack that ships with Rails 8 (Minitest + system tests via Capybara/Selenium). No swap to RSpec in this change.
- Add `bin/setup` so a fresh clone runs end-to-end (`bundle install`, `db:prepare`, asset prep) on a developer machine.
- Add a minimal CI workflow (GitHub Actions) that runs `bin/rubocop`, `bin/brakeman`, and `bin/rails test` against PostgreSQL.
- Add a brief `README.md` covering local setup; remove the stock Rails README boilerplate.
- Update `.gitignore` to exclude `log/`, `tmp/`, `storage/`, `.env*`, and credential keys.
- Wire a placeholder root route (`HomeController#index`) returning a stub page so the app boots green.

Non-goals (explicit, to keep this change small):
- No Donor / Donation / lookup-table models. Those land in a follow-up change.
- No data migration from the existing `.accdb` file.
- No production deploy config (Kamal, hosting, domains).
- No styling system beyond the Rails 8 defaults (Propshaft + import maps + Tailwind is a later decision).

## Capabilities

### New Capabilities
- `app-foundation`: The Rails application itself — boot, configuration, dependency baseline, dev/test tooling, CI, and the placeholder root route.
- `authentication`: User accounts, sessions, and password reset flows generated by Rails 8's authentication scaffold. The capability ships wired but with no users seeded and no controllers yet protected.

## Impact

- **Repository layout**: introduces the standard Rails directory tree (`app/`, `config/`, `db/`, `bin/`, `test/`) at the repo root, alongside the existing `docs/` and `openspec/` directories.
- **Tooling required on contributor machines**: Ruby (version pinned by `.ruby-version`), PostgreSQL 14+, Node.js only if/when JS bundling is later added (not in this change).
- **CI**: a new `.github/workflows/ci.yml` runs lint, security scan, and tests on every push and PR.
- **Downstream changes**: every subsequent OpenSpec change (donor model, donation model, reports, tax-receipt mailer, etc.) depends on this scaffold being merged first.
- **No production systems are affected** — the charity continues to use Access until later changes deliver migration tooling.
